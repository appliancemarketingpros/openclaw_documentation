# `openclaw logs`

Отслеживание файловых журналов Gateway через RPC (работает в удаленном режиме).

Связанные разделы:

  * Обзор журналирования: [Журналирование](</ru/logging>)
  * CLI Gateway: [gateway](</ru/cli/gateway>)


## Параметры

  * `--limit <n>`: максимальное количество возвращаемых строк журнала (по умолчанию `200`)
  * `--max-bytes <n>`: максимальное количество байтов для чтения из файла журнала (по умолчанию `250000`)
  * `--follow`: следить за потоком журнала
  * `--interval <ms>`: интервал опроса при слежении (по умолчанию `1000`)
  * `--json`: выводить события JSON с разделением по строкам
  * `--plain`: вывод обычного текста без стилизованного форматирования
  * `--no-color`: отключить цвета ANSI
  * `--local-time`: отображать метки времени в вашем локальном часовом поясе (по умолчанию)
  * `--utc`: отображать метки времени в UTC


## Общие параметры RPC Gateway

`openclaw logs` также принимает стандартные флаги клиента Gateway:

  * `--url <url>`: URL WebSocket Gateway
  * `--token <token>`: токен Gateway
  * `--timeout <ms>`: тайм-аут в мс (по умолчанию `30000`)
  * `--expect-final`: ждать финального ответа, когда вызов Gateway выполняется через агента


Когда вы передаете `--url`, CLI не применяет учетные данные из конфигурации или окружения автоматически. Явно укажите `--token`, если целевой Gateway требует аутентификации.

## Примечания

  * По умолчанию метки времени отображаются в вашем локальном часовом поясе. Используйте `--utc` для вывода в UTC.
  * Если неявный Gateway local loopback запрашивает сопряжение, закрывается во время подключения или истекает по тайм-ауту до ответа `logs.tail`, `openclaw logs` автоматически переключается на настроенный файловый журнал Gateway. Явные цели `--url` не используют этот резервный вариант.
  * `openclaw logs --follow` не продолжает следить за резервными настроенными файлами после сбоев неявного локального RPC Gateway. В Linux он использует активный пользовательский журнал Gateway в systemd по PID, когда он доступен, и выводит выбранный источник журнала; в противном случае он продолжает повторять попытки обращения к текущему Gateway вместо отслеживания потенциально устаревшего соседнего файла.
  * При использовании `--follow` временные отключения gateway (закрытие WebSocket, тайм-аут, разрыв соединения) запускают автоматическое переподключение с экспоненциальной задержкой (до 8 повторных попыток, с ограничением 30 с между попытками). При каждой повторной попытке в stderr выводится предупреждение, а после успешного опроса выводится уведомление `[logs] gateway reconnected`. В режиме `--json` и предупреждение о повторной попытке, и переход к повторному подключению выводятся как записи `{"type":"notice"}` в stderr. Неустранимые ошибки (сбой аутентификации, неверная конфигурация) по-прежнему немедленно завершают выполнение.
